# 项目风险管理技能

> 版本: 1.0
> 系统化的项目风险识别、评估和应对方法

## 🎯 核心目标

通过系统的风险管理：
- ✅ 预见潜在问题，提前准备
- ✅ 降低项目延期概率
- ✅ 确保项目结果达到预期
- ✅ 提高项目成功率

---

## 📚 第一部分：风险识别（5W1H法）

### 1.1 风险识别维度

#### 技术风险（最常见）
```
⚠️ 典型问题：
- 技术选型不成熟
- 第三方服务不稳定
- 技术难点预估不足
- 兼容性问题
- 性能瓶颈

✅ 识别问题：
1. 这个技术团队熟悉吗？
2. 有成功案例吗？
3. 需要外部依赖吗？
4. 有备选方案吗？
```

#### 资源风险（最容易导致延期）
```
⚠️ 典型问题：
- 关键人员离职
- 人员技能不足
- 资源分配冲突
- 外包团队配合差
- 预算不足

✅ 识别问题：
1. 关键岗位有备份吗？
2. 成员技能匹配吗？
3. 有多项目冲突吗？
4. 预算有缓冲吗？
```

#### 时间风险（最直接）
```
⚠️ 典型问题：
- 任务拆分太粗
- 依赖关系不清
- 测试时间不足
- 审批流程长
- 里程碑不合理

✅ 识别问题：
1. 每个任务有具体产出吗？
2. 依赖关系画出来了吗？
3. 预留了测试时间吗？
4. 里程碑可验证吗？
```

#### 范围风险（最隐蔽）
```
⚠️ 典型问题：
- 需求变更频繁
- 需求理解偏差
- 范围蔓延
- 干系人期望不一致
- 需求优先级不清

✅ 识别问题：
1. 需求文档完整吗？
2. 变更流程明确吗？
3. 干系人期望对齐了吗？
4. 优先级排好了吗？
```

#### 质量风险（影响最大）
```
⚠️ 典型问题：
- 测试覆盖不足
- 性能指标不达标
- 用户体验差
- 数据安全问题
- 系统稳定性差

✅ 识别问题：
1. 有测试计划吗？
2. 性能标准明确吗？
3. 安全测试做了吗？
4. 灾备方案有吗？
```

#### 外部风险（不可控但可缓解）
```
⚠️ 典型问题：
- 市场环境变化
- 政策法规调整
- 竞争对手动作
- 自然灾害
- 供应商问题

✅ 识别问题：
1. 外部因素监控了吗？
2. 有应急预案吗？
3. 供应商备份了吗？
```

### 1.2 风险识别工具

#### 头脑风暴法
```
步骤：
1. 召集团队成员
2. 列出所有可能的风险
3. 不批判，只记录
4. 后续分类整理

规则：
- 每个人至少提出3个风险
- 鼓励"疯狂"的想法
- 记录所有想法
```

#### 检查清单法
```
技术类：
□ 技术选型是否成熟？
□ 团队是否有相关经验？
□ 是否有技术难点？
□ 是否有外部依赖？

资源类：
□ 人员是否到位？
□ 预算是否充足？
□ 设备是否就绪？
□ 外部资源是否可靠？

时间类：
□ 计划是否合理？
□ 里程碑是否可验证？
□ 依赖关系是否清晰？
□ 是否预留缓冲？

范围类：
□ 需求是否明确？
□ 变更流程是否建立？
□ 优先级是否确定？
□ 干系人是否对齐？

质量类：
□ 测试计划是否完整？
□ 验收标准是否明确？
□ 性能要求是否达标？
□ 安全措施是否到位？
```

#### 专家访谈法
```
访谈对象：
- 领域专家
- 项目经理
- 技术负责人
- 有类似项目经验的人

访谈问题：
1. 这个项目最大的风险是什么？
2. 哪些环节最容易出问题？
3. 有没有类似项目的经验？
4. 应该特别关注什么？
```

#### 历史数据分析
```
数据来源：
- 以前项目的风险清单
- 延期项目的复盘报告
- 失败案例分析
- 行业风险统计

分析要点：
1. 哪类风险最常见？
2. 哪类风险影响最大？
3. 哪类风险最难应对？
```

## 📊 第二部分：风险评估（矩阵法）

### 2.1 风险评估矩阵

```
                    影响
            低    中    高
        1    2    3
可能性
中      2    4    6   ← 关键风险
        3    6    9   ← 严重风险
```

### 2.2 风险等级定义

#### 可能性（1-3分）
```
1分（低）:
- 发生概率 < 20%
- 从未见过类似情况
- 可控因素多

2分（中）:
- 发生概率 20% - 60%
- 偶尔见到类似情况
- 部分可控

3分（高）:
- 发生概率 > 60%
- 经常见到类似情况
- 难以控制
```

#### 影响程度（1-3分）
```
1分（低）:
- 影响范围小
- 影响程度轻
- 容易恢复
- 成本增加 < 10%

2分（中）:
- 影响范围中等
- 影响程度中等
- 需要一定时间恢复
- 成本增加 10% - 30%

3分（高）:
- 影响范围大
- 影响程度严重
- 很难恢复
- 成本增加 > 30%
- 可能导致项目失败
```

#### 风险等级（9分制）
```
1-3分：低风险 🟢
- 监控即可，无需特别应对

4-6分：中风险 🟡
- 制定应对计划，定期检查

7-9分：高风险 🔴
- 重点关注，立即制定应对措施
- 必要时调整项目计划
```

### 2.3 评估示例

```
风险1：关键人员离职
可能性：2分（中等 - 偶尔发生）
影响程度：3分（高 - 严重影响）
风险等级：6分（中风险 🟡）
优先级：高（因为影响大）

风险2：技术选型不成熟
可能性：1分（低 - 从未发生过）
影响程度：3分（高 - 严重影响）
风险等级：3分（低风险 🟢）
优先级：中（虽然等级低，但影响大）

风险3：需求变更频繁
可能性：3分（高 - 经常发生）
影响程度：2分（中 - 中等影响）
风险等级：6分（中风险 🟡）
优先级：高（因为可能性和影响都大）

风险4：第三方服务不稳定
可能性：2分（中等）
影响程度：2分（中等）
风险等级：4分（中风险 🟡）
优先级：中
```

## 🛡️ 第三部分：风险应对策略

### 3.1 四大应对策略

#### 规避（Avoid）
```
适用情况：
- 风险等级高
- 影响严重
- 成本可接受

具体措施：
- 放弃有风险的功能
- 选择成熟技术
- 增加人员备份
- 缩小项目范围

示例：
- 如果技术不成熟，选择替代方案
- 如果人手不足，减少功能
- 如果时间紧，降低质量标准
```

#### 缓解（Mitigate）
```
适用情况：
- 风险等级中高
- 无法完全规避
- 可以降低影响

具体措施：
- 增加测试
- 提前培训
- 预留时间
- 制定预案

示例：
- 技术风险 → 提前技术调研
- 人员风险 → 提前技能培训
- 时间风险 → 预留缓冲时间
- 质量风险 → 增加测试资源
```

#### 转移（Transfer）
```
适用情况：
- 风险等级中
- 可以委托他人
- 成本可接受

具体措施：
- 外包
- 购买保险
- 签订合同
- 使用第三方服务

示例：
- 部分功能外包给专业公司
- 购买云服务的SLA保障
- 与供应商签订保障条款
```

#### 接受（Accept）
```
适用情况：
- 风险等级低
- 应对成本高
- 影响可接受

具体措施：
- 监控跟踪
- 制定应急计划
- 预留预算

示例：
- 小功能缺陷，后续迭代修复
- 轻微性能问题，可优化时处理
- 预算小超支，接受
```

### 3.2 常见风险应对模板

#### 技术风险应对
```
风险：技术选型不成熟
应对策略：规避/缓解

措施：
1. 提前进行技术调研（2周）
2. 做POC验证（1周）
3. 准备备选方案（2个）
4. 咨询技术专家

责任人：技术总监
时间节点：项目启动前
成功标准：技术验证通过
```

#### 资源风险应对
```
风险：关键人员离职
应对策略：缓解

措施：
1. 建立知识文档（持续）
2. 人员备份（至少1个备选）
3. 技术交叉培训（每周）
4. 制定交接流程（文档化）

责任人：项目经理
时间节点：项目全过程
成功标准：离职影响 < 3天
```

#### 时间风险应对
```
风险：任务延期
应对策略：缓解

措施：
1. 任务拆分到3-5天
2. 每日站会跟踪
3. 预留20%缓冲时间
4. 关键路径优先

责任人：项目经理 + 技术负责人
时间节点：项目全过程
成功标准：延期 < 10%
```

#### 范围风险应对
```
风险：需求变更
应对策略：缓解

措施：
1. 建立变更流程（文档化）
2. 变更影响评估（3天）
3. 优先级排序（干系人确认）
4. 版本控制（功能冻结）

责任人：产品经理
时间节点：项目全过程
成功标准：变更受控
```

## 📋 第四部分：风险跟踪与复盘

### 4.1 风险跟踪机制

#### 定期检查
```
频率：
- 高风险：每周检查
- 中风险：每两周检查
- 低风险：每月检查

内容：
- 风险状态变化
- 应对措施执行情况
- 新增风险
- 风险触发事件

输出：
- 风险状态报告
- 应对措施更新
- 新风险清单
```

#### 里程碑检查
```
时机：
- 每个里程碑完成时
- 项目阶段转换时
- 重大变更发生时

内容：
- 回顾已识别风险
- 评估新风险
- 更新风险登记册
- 调整应对策略
```

### 4.2 风险复盘方法

#### 触发事件复盘
```
时机：风险触发时

问题：
1. 为什么发生？
2. 为什么没提前识别？
3. 为什么应对措施失效？
4. 下次如何避免？

输出：
- 事故报告
- 经验教训
- 流程改进
```

#### 项目复盘
```
时机：项目结束后

内容：
1. 有哪些风险发生了？
2. 有哪些风险没发生？
3. 哪些应对措施有效？
4. 哪些应对措施无效？
5. 有哪些新的经验教训？

输出：
- 风险总结报告
- 最佳实践
- 流程优化
```

## 🎓 第五部分：实战案例

### 案例1：软件开发项目

#### 项目背景
```
- 周期：3个月
- 团队：5人
- 功能：20个
- 复杂度：中等
```

#### 风险识别
```
技术风险：
- 第三方API不稳定（可能性2，影响2）→ 中风险
- 性能要求高（可能性2，影响3）→ 中风险

资源风险：
- 关键开发人员可能离职（可能性1，影响3）→ 低风险
- 测试人员不足（可能性2，影响2）→ 中风险

时间风险：
- 需求不明确导致返工（可能性3，影响2）→ 中风险
- 依赖外部系统（可能性2，影响2）→ 中风险

范围风险：
- 需求变更频繁（可能性3，影响2）→ 中风险
- 功能蔓延（可能性2，影响2）→ 中风险
```

#### 应对计划
```
高优先级：
1. 需求变更 → 建立变更流程，冻结期
2. 外部依赖 → 提前联调，备选方案
3. 需求不明确 → 需求评审，原型确认

中优先级：
1. 第三方API → 做接口mock，监控
2. 测试人员不足 → 自动化测试，交叉测试
3. 性能要求 → 提前性能测试，优化
```

#### 结果
```
实际发生：
- 需求变更3次（缓解成功）
- 第三方API延期1周（备选方案生效）
- 测试人员不足（自动化测试覆盖）

结果：
- 项目延期1周（原计划3个月）
- 在可接受范围内
```

### 案例2：系统集成项目

#### 项目背景
```
- 周期：6个月
- 团队：10人
- 系统：5个
- 复杂度：高
```

#### 风险识别
```
技术风险：
- 系统兼容性问题（可能性2，影响3）→ 中风险
- 数据迁移风险（可能性2，影响3）→ 中风险

资源风险：
- 关键人员离职（可能性2，影响3）→ 中风险
- 外包团队配合（可能性3，影响2）→ 中风险

时间风险：
- 多系统协调（可能性2，影响3）→ 中风险
- 集成测试时间不足（可能性2，影响2）→ 中风险

范围风险：
- 需求理解偏差（可能性2，影响3）→ 中风险
- 干系人期望不一致（可能性2，影响2）→ 中风险
```

#### 应对计划
```
关键措施：
1. 兼容性 → 建立测试环境，提前验证
2. 数据迁移 → 做迁移测试，备份方案
3. 外包团队 → 建立SLA，定期检查

5. 需求偏差 → 原型演示，定期评审
```

#### 结果
```
实际发生：
- 数据迁移失败1次（备份恢复）
- 外包团队延期2周（SLA生效，补偿）
- 集成问题3个（分阶段解决）

结果：
- 项目延期2周（原计划6个月）
- 在可接受范围内
```

## 🚀 第六部分：快速启动指南

### 步骤1：启动会（第1周）
```
1. 团队头脑风暴（2小时）
2. 识别所有风险（不限数量）
3. 分类整理
4. 录入风险登记册
```

### 步骤2：评估风险（第1周）
```
1. 评估每个风险的可能性和影响
2. 计算风险等级
3. 标注高、中、低风险
4. 确定优先级
```

### 步骤3：制定应对计划（第1周）
```
1. 高风险：立即制定详细应对计划
2. 中风险：制定基本应对计划
3. 低风险：监控即可
4. 分配责任人
```

### 步骤4：定期检查（持续）
```
1. 高风险：每周检查
2. 中风险：每两周检查
3. 低风险：每月检查
4. 更新风险状态
```

### 步骤5：复盘总结（项目结束）
```
1. 哪些风险发生了？
2. 哪些应对措施有效？
3. 有哪些经验教训？
4. 更新风险知识库
```

## 💡 最佳实践

### ✅ 要做的

1. **提前识别**
   - 项目启动时就开始识别
   - 持续更新风险清单
   - 定期重新评估

2. **全员参与**
   - 每个团队成员都有责任
   - 鼓励上报风险
   - 不惩罚识别风险的人

3. **文档化**
   - 所有风险记录在案
   - 应对措施明确可执行
   - 更新历史可追溯

4. **定期回顾**
   - 每个里程碑都要回顾
   - 项目结束后复盘
   - 更新风险知识库

5. **持续改进**
   - 从每次风险中学习
   - 优化风险管理流程
   - 分享最佳实践

### ❌ 不要做的

1. **忽视低风险**
   - 低风险也可能升级
   - 持续监控很重要
   - 条件变化时重新评估

2. **只记录不行动**
   - 识别风险只是第一步
   - 要制定应对措施
   - 跟踪执行情况

3. **风险登记册一成不变**
   - 风险会变化
   - 新风险会出现
   - 定期更新

4. **忽视干系人**
   - 干系人期望也是风险
   - 定期沟通对齐
   - 管理期望

5. **只看风险不看机会**
   - 风险背后有机会
   - 把风险转化为机会
   - 建立创新文化

---

## 🎯 核心要点总结

```
风险管理的本质：
- 不是消除所有风险（不可能）
- 而是识别、评估、应对（可控）
- 把不可控变成可控

风险管理的关键：
- 提前识别（越早越好）
- 持续跟踪（不能松懈）
- 及时应对（快速响应）

风险管理的目标：
- 降低风险发生的可能性
- 减轻风险发生的影响
- 确保项目成功交付
```

---

**记住：风险管理不是为了消灭风险，而是为了管理风险，让风险可控。**
