## 🎯 为什么要管理风险？

```
不管理风险 → 问题频发 → 延期加班 → 结果不理想
      ↓
管理风险   → 预见问题   → 提前准备   → 项目成功
```

## 📊 风险评估方法

### 可能性评估（1-3分）

**1分（低）**
- 发生概率 < 20%
- 从未见过类似情况

**2分（中）**
- 发生概率 20% - 60%
- 偶尔见到类似情况

**3分（高）**
- 发生概率 > 60%
- 经常见到类似情况

### 影响程度评估（1-3分）

**1分（低）**
- 影响范围小，容易恢复
- 成本增加 < 10%

**2分（中）**
- 影响范围中等，需要一定时间恢复
- 成本增加 10% - 30%

**3分（高）**
- 影响范围大，很难恢复
- 成本增加 > 30%，可能导致项目失败

### 风险等级计算

```
风险等级 = 可能性 × 影响程度

1-3分：低风险 🟢  → 监控即可
4-6分：中风险 🟡  → 制定应对计划
7-9分：高风险 🔴  → 立即采取措施
```

## 🛡️ 风险应对策略

### 规避（Avoid）- 彻底避免
```
何时用：高风险 + 高影响
怎么做：
- 放弃有风险的功能
- 选择成熟技术
- 增加人员备份

示例：
- 如果技术不成熟，选择替代方案
- 如果人手不足，减少功能
```

### 缓解（Mitigate）- 降低影响
```
何时用：中高风险
怎么做：
- 增加测试
- 提前培训
- 预留时间
- 制定预案

示例：
- 技术风险 → 提前技术调研
- 人员风险 → 提前技能培训
- 时间风险 → 预留缓冲时间
```

### 转移（Transfer）- 让别人承担
```
何时用：中风险
怎么做：
- 外包
- 购买保险
- 签订合同

示例：
- 部分功能外包给专业公司
- 购买云服务的SLA保障
- 与供应商签订保障条款
```

### 接受（Accept）- 承担后果
```
何时用：低风险
怎么做：
- 监控跟踪
- 制定应急计划
- 预留预算

示例：
- 小功能缺陷，后续迭代修复
- 轻微性能问题，可优化时处理
- 预算小超支，接受
```

## 🔍 如何识别风险？

### 6大风险类别

#### 1. 技术风险 ⚠️
```
典型问题：
- 技术选型不成熟
- 第三方服务不稳定
- 兼容性问题
- 性能瓶颈

关键问题：
✓ 团队熟悉这个技术吗？
✓ 有成功案例吗？
✓ 有备选方案吗？
```

#### 2. 资源风险 👥
```
典型问题：
- 关键人员离职
- 人员技能不足
- 资源分配冲突
- 预算不足

关键问题：
✓ 关键岗位有备份吗？
✓ 成员技能匹配吗？
✓ 预算有缓冲吗？
```

#### 3. 时间风险 ⏰
```
典型问题：
- 任务拆分太粗
- 依赖关系不清
- 测试时间不足
- 审批流程长

关键问题：
✓ 任务有具体产出吗？
✓ 依赖关系清楚吗？
✓ 预留了缓冲时间吗？
```

#### 4. 范围风险 📏
```
典型问题：
- 需求变更频繁
- 需求理解偏差
- 范围蔓延
- 期望不一致

关键问题：
✓ 需求文档完整吗？
✓ 变更流程明确吗？
✓ 期望对齐了吗？
```

#### 5. 质量风险 ✅
```
典型问题：
- 测试覆盖不足
- 性能不达标
- 用户体验差
- 数据安全问题

关键问题：
✓ 有测试计划吗？
✓ 性能标准明确吗？
✓ 安全测试做了吗？
```

#### 6. 外部风险 🌍
```
典型问题：
- 市场环境变化
- 政策法规调整
- 竞争对手动作
- 供应商问题

关键问题：
✓ 外部因素监控了吗？
✓ 有应急预案吗？
✓ 供应商备份了吗？
```

## ❓ 常见问题

**Q: 风险管理会浪费时间吗？**
A: 相比于延期和返工，风险管理的投入非常值得。它能帮你节省大量时间。

**Q: 小项目也需要风险管理吗？**
A: 需要！大小项目都有风险，小项目可能资源更少，更需要提前识别风险。

**Q: 如何确定风险的可能性？**
A: 参考历史数据、团队能力、外部因素，结合经验判断。

**Q: 风险太多管不过来怎么办？**
A: 优先关注高风险，中风险制定基本计划，低风险只需监控。

**Q: 风险触发后怎么办？**
A: 立即执行应对措施，记录触发事件，更新风险状态，复盘总结。
